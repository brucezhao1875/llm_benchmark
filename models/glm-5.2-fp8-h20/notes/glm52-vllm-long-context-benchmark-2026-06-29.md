# GLM-5.2-FP8 长上下文并发压测报告

> 测试日期：2026-06-29
> 测试机器：h20-02
> 模型：GLM-5.2-FP8
> 推理框架：vLLM 0.23.0
> GPU：8 x NVIDIA H20-3e
> 部署方式：systemd service
> 测试性质：长上下文 prefill / 并发排队能力验证，不是最终生产容量报告

## 1. 本次测试目标

本次测试目标是观察 GLM-5.2-FP8 在不同上下文长度和并发数下的基础表现。

测试维度：

| 维度 | 档位 |
|---|---|
| 上下文长度 | 32K / 64K / 128K / 256K |
| 并发数 | 1 / 2 / 4 / 8 / 16 |
| 输出长度 | `max_tokens=16` |
| 请求方式 | OpenAI-compatible `chat/completions`，streaming |

重点观察指标：

| 指标 | 含义 |
|---|---|
| TTFT | 从请求发出到首个输出 token 返回的时间，长上下文下主要反映 prefill 成本和排队时间 |
| Latency | 请求总耗时 |
| QPS | 每秒完成请求数 |
| Aggregate output TPS | 服务整体输出 token/s，本次输出很短，因此主要作为参考 |
| Decode TPS | 首 token 后的输出速度，长上下文测试中不是主要瓶颈 |
| Token 间隔 | streaming 输出中相邻 token/chunk 的间隔 |
| 失败率 | 请求是否被拒绝、超时或报错 |

## 3. 测试方法

### 3.1 请求配置

API：

```text
http://127.0.0.1:8000/v1/chat/completions
```

请求核心参数：

```json
{
  "model": "glm-5.2-fp8",
  "temperature": 0,
  "max_tokens": 16,
  "stream": true,
  "stream_options": {
    "include_usage": true
  }
}
```

说明：

```text
max_tokens=16 是为了把测试重点放在长上下文 prefill，而不是长输出 decode。
因此本报告的 output TPS 不能代表模型在长输出场景下的最大生成吞吐。
```

### 3.2 Prompt 构造方式

使用 tokenizer 构造接近目标长度的中文填充文本。

完成的实际 prompt token 规模：

| 目标上下文 | 实际基础 prompt tokens | API usage 中平均 prompt tokens |
|---:|---:|---:|
| 32K | 32752 | 约 32788-32789 |
| 64K | 65520 | 约 65555-65556 |
| 128K | 131056 | 约 131091-131093 |
| 256K | 261674 | 约 261709 |

注意：

```text
256K 档不能直接构造 262144 tokens 的 prompt。
因为 max_model_len=262144 是输入 + 输出的总上限。
请求里还有唯一请求前缀、chat template、特殊 token 和 max_tokens=16。
第一次 256K 测试因超过上限返回 400 Bad Request。
随后将基础 prompt 降到约 261674 tokens 后重测成功。
```

### 3.3 请求数量

每个组合请求数等于并发数，即每档只打一轮：

| 并发 | 请求数 |
|---:|---:|
| 1 | 1 |
| 2 | 2 |
| 4 | 4 |
| 8 | 8 |
| 16 | 16 |

原因：

```text
长上下文请求耗时很长，尤其 128K / 256K 下单轮已经达到数分钟到十几分钟。
本次目标是得到初步容量曲线，而不是长时间稳定性报告。
```

## 6. 关键观察

### 6.1 长上下文 TTFT 基本随输入长度线性增长

单并发下 TTFT：

| 上下文 | 单请求 TTFT |
|---:|---:|
| 32K | 9.94s |
| 64K | 19.83s |
| 128K | 43.04s |
| 256K | 101.40s |

趋势：

```text
32K -> 64K：约 2.0x
64K -> 128K：约 2.2x
128K -> 256K：约 2.4x
```

说明：

```text
长上下文场景主要瓶颈是 prefill。
上下文越长，首 token 延迟越高。
256K 单请求首 token 已经超过 100 秒，不适合作为普通在线交互默认配置。
```

### 6.2 并发增加没有提高长上下文 QPS

32K 下 QPS：

```text
并发 1：0.098
并发 2：0.104
并发 4：0.105
并发 8：0.106
并发 16：0.103
```

64K 下 QPS：

```text
基本稳定在 0.05 req/s
```

128K 下 QPS：

```text
基本稳定在 0.023 req/s
```

256K 下 QPS：

```text
基本稳定在 0.010 req/s
```

这说明：

```text
长上下文 prefill 很重。
在这些超长输入下，提高并发主要造成排队，不能明显提高请求吞吐。
```

### 6.3 并发越高，TTFT 和 p95 延迟越差

以 128K 为例：

| 并发 | 平均 TTFT | p95 延迟 |
|---:|---:|---:|
| 1 | 43.04s | 43.18s |
| 2 | 65.61s | 85.94s |
| 4 | 108.96s | 171.31s |
| 8 | 194.06s | 341.14s |
| 16 | 364.19s | 681.15s |

结论：

```text
对长上下文请求，系统更像是在按 prefill 成本排队。
并发越高，排队越明显，p95 延迟近似按并发倍数拉长。
```

### 6.4 Decode 不是本次瓶颈

本次输出只有：

```text
max_tokens=16
```

所以长上下文请求的总耗时几乎都在：

```text
prefill + 排队
```

不是 decode。

在单请求下，decode token 间隔仍然较低：

```text
32K：17.13ms
64K：17.69ms
128K：14.44ms
256K：17.16ms
```

这说明一旦 prefill 完成，短输出阶段并不慢。

### 6.5 256K 可用，但不适合高并发在线服务

256K 单请求：

```text
TTFT ≈ 101s
Latency ≈ 101s
```

256K 并发 8：

```text
p95 latency ≈ 801s，约 13.4 分钟
```

256K 并发 16：

```text
预计会接近 1600s，约 26-27 分钟
本次主动中止
```

结论：

```text
256K 可以作为专用长文档/仓库级任务能力，但不适合作为普通用户在线交互服务。
```

## 7. 对部署策略的影响

### 7.1 不建议默认服务使用 256K

如果把主服务默认开放到 256K：

```text
短请求也能跑
但长请求会占用大量 prefill 时间
高并发长请求会让队列和 p95 延迟急剧恶化
需要强限流和调度隔离
```

### 7.2 推荐按服务类型拆分

建议拆成多类服务：

| 服务类型 | max_model_len | 用途 |
|---|---:|---|
| 在线通用服务 | 32768 | 普通问答、短 RAG、基础 agent |
| Agent 增强服务 | 65536 或 131072 | 工具调用、多轮任务、较长上下文 |
| 长上下文专用服务 | 262144 | 长文档、代码仓库、低并发任务 |

### 7.3 Gateway 必须做上下文限流

Gateway 层建议控制：

```text
最大 input tokens
最大 max_tokens
按模型实例路由
按上下文长度分流
长上下文请求限并发
长上下文请求单独队列
租户级限额
超长请求异步化
```

建议策略：

| 输入长度 | 路由建议 |
|---:|---|
| <= 32K | 默认在线实例 |
| 32K-128K | Agent/长上下文实例，限制并发 |
| 128K-256K | 专用长任务实例，建议异步任务化 |
| > 256K | 当前服务不接受，或切换 1M 专用实验服务 |

## 8. 对业务 SLA 的建议

按本次数据，粗略建议：

| 场景 | 建议上下文 | 建议并发 | 说明 |
|---|---:|---:|---|
| 在线聊天 | <= 32K | 低到中等 | 32K 单请求 TTFT 约 10s，仍偏高，短 prompt 会更快 |
| RAG 问答 | <= 32K / 64K | 控制在 1-4 | 64K 单请求 TTFT 约 20s |
| Agent 长任务 | 64K / 128K | 控制在 1-2 | 128K 并发 2 p95 延迟约 86s |
| 长文档分析 | 128K / 256K | 建议并发 1 | 256K 单请求 TTFT 约 101s |
| 256K 高并发 | 不建议 | 不建议 | 并发 8 p95 已约 13 分钟 |

## 9. 测试局限

本次测试有以下限制：

```text
每个组合只打一轮，请求数量少。
输出只有 16 tokens，不能代表长输出 decode 吞吐。
prompt 是重复填充文本，不能完全代表真实长文档 attention 行为。
没有同步采集 GPU util、KV cache 使用率和 vLLM 内部队列指标。
没有通过 Gateway，未包含网关开销。
没有做 1M 上下文。
256K 并发 16 被主动中止，没有完成数据。
```

因此本报告适合用于：

```text
判断长上下文 prefill 延迟量级
判断高并发长上下文是否可用
制定 Gateway 限流和服务拆分策略
```

不适合直接作为：

```text
最终生产 SLA
最大容量验收报告
完整性能白皮书
```

## 10. 结论

1. 当前 GLM-5.2-FP8 vLLM 服务已成功以 `max_model_len=262144` 启动，并完成 32K/64K/128K/256K 大部分组合测试。
2. 长上下文下，主要瓶颈是 prefill，TTFT 随输入长度近似线性增长。
3. 32K 单请求 TTFT 约 10s，64K 约 20s，128K 约 43s，256K 约 101s。
4. 对长上下文请求，提高并发不会显著提升 QPS，主要表现为排队和 p95 延迟上升。
5. 256K 上下文可用，但只适合低并发、专用长任务，不适合作为默认在线服务。
6. 建议生产上拆分 32K 在线实例、64K/128K Agent 实例、256K 长上下文专用实例。
7. Gateway 必须按 input tokens 做路由、限流和配额控制，否则少量超长请求就可能拖垮整体 p95。
